# Gigahorse Benchmarks — YulTracer Analysis

> See the main [`README`](../../../README.md) for build instructions and [`GUIDE`](../../../GUIDE.md) for setting up new projects.

This directory contains benchmarks originally sourced from the [gigahorse-benchmarks](https://github.com/nevillegrech/gigahorse-benchmarks), along with a collection of processed variants adapted for analysis with **YulTracer** (abbreviated **yult**), our tool for analysing smart contracts compiled into EVM-flavoured Yul.

---

## Quick Start

This directory contains a YulTracer-compatible subset of the gigahorse-benchmarks. Each example:

* Starts from vulnerable source in the gigahorse-benchmarks
* Is transformed to be Solidity 0.8.x compatible (while preserving semantics)
* Is compiled to Yul
* Is analysed using YulTracer

All necessary **preprocessing of source files has already been performed**. You can immediately run YulTracer on any processed example by entering its directory and running:

```bash
make          # compile
make run      # analyse
```

To run YulTracer across all benchmarks, see [Running YulTracer on All Benchmarks](#running-yultracer-on-all-benchmarks). To compare against other tools, see [Comparing YulTracer with Other Tools](#comparing-yultracer-with-other-tools).

## Directory Structure

```
gigahorse-benchmarks/
├── original-vulnerable-sources/
├── reentr-assert-vuln/
├── reentr-assert-safe/
├── reentr-vuln/
└── ac-vuln/
```

### `original-vulnerable-sources/`

This directory contains the **109 original Solidity source files** from the gigahorse-benchmarks. Each file is an unmodified Solidity source for an example demonstrating a vulnerability, e.g., `reentrancy_0x23a91059fdc9579a9fbd0edc5f2ea0bfdb70deb4.sol`.

### `reentr-assert-vuln/` and `reentr-assert-safe/`

These directories contain processed versions of the reentrancy examples from the original set, **with assertions added**, adapted for analysis with YulTracer and other tools. The `-vuln` variant contains vulnerable contracts; the `-safe` variant contains their safe counterparts.

### `reentr-vuln/`

Processed reentrancy examples **without assertions**, adapted for YulTracer analysis.

### `ac-vuln/`

Processed access-control vulnerability examples, adapted for YulTracer analysis.

Each entry in these directories corresponds to a specific benchmark and is named after the original Solidity file, but without the `.sol` extension. For example, for the file `original-vulnerable-sources/reentrancy_etherbank.sol` we would have a corresponding directory:

```
reentr-assert-vuln/
└── reentrancy_etherbank/
```

Each such directory contains the following:

#### `abi/`

Contains the ABI for the main contract (entry point) in the processed source. For instance:

* `EtherBank_pruned.json`: ABI of `EtherBank.sol`. The `_pruned` suffix refers to the possibility of ABI pruning by preprocessing scripts; currently, these are unpruned and complete.

#### `scripts/`

Contains the scripts used to transform the original benchmark into a form suitable for analysis. Notably:

* `transform_solidity_040_to_080.py`: Converts legacy Solidity versions (e.g., 0.4.x) to 0.8.x-compatible syntax and constructs. Each example may have its own such script specifically tailored to handle it. Further manual inspection may have been performed to ensure compatibility; where necessary, manual modifications will have a corresponding comment in the processed source file indicating what was done.

#### `src/`

Contains the solidity sources for the example in the current directory:
* `<original>.sol`: The original file, copied from `original-vulnerable-sources/`
* `<EntryPoint>.sol`: A transformed Solidity file generated by scripts and possibly by further manual inspection, containing the entry-point contract.

For example:

```
src/
├── reentrancy_etherbank.sol        # original source: contains entry-point contract `EtherBank`
└── EtherBank.sol                   # transformed source: named after entry-point contract
```

The filename must match the main contract’s name (`EtherBank` in this case).

#### `yul/`

Generated output after building the example with the included `Makefile`.

This contains:

* The `.yul` files generated by `solc`
* JSON artefacts generated during compilation
* Post-processed the entry-point Yul with YulTracer-specific passes applied:

    * **Linking**: Resolving any linker symbols
    * ****yult** activation**: Enables internal YulTracer functions
    * ****Yult** object rewriting**: Normalizes internal objects
    * **Unchecked arithmetic disabling**: To preserve semantics from pre-0.8 Solidity
    * **ABI object name fixing**: Ensures the object names in the ABI (e.g., `EtherBank_*`) match the compiled Yul output

#### `Makefile`

Each directory has a `Makefile` that controls the full build and analysis pipeline. e.g.:

Key parameters:

```make
ENTRY_OBJECT_NAME := EtherBank
```

This defines the contract name and is used to:

* Select the entry-point contract to analyse (`EtherBank` in this case)
* Name the output `.sol` file to be compiled (`EtherBank.sol` in this case)

```make
CORE_ARGS ?= \
    -g \
    -b 2 \
    -o-call-stack 2 \
    -deploy-value 0 \
    -o-uint-domain "1000000000000000001" \
    -o-default-spending 1000000000000000001 \
    -max-time 1
```

These are the flags/parameters passed to `yult`. A typical direct invocation of `yult` may look like:

```bash
dune exec -- yult -i <input.yul> $(CORE_ARGS)
```
Common parameters used in these analyses:

* `-o-default-spending` and `-o-uint-domain` (opponent spending and opponent `uint` arguments): 

    - 1 (1 WEI)
    - 1000000000000000000 (1 ETH)
    - 1000000000000000001 (>1 ETH)
    - 2000000000000000000 (2 ETH)
    - 2000000000000000001 (>2 ETH)

* `-o-call-stack` (depth at which o-calls can occur): 2, 3, 4
* `-no-wait` (if no timestamps are used in the contract, i.e. time isn't important)
* `-max-time` (maximum number of cumulative days the opponent can wait): 1 (unless set, opponent will wait 7 days by default, so setting `max-time` to 1 will have the opponent wait once, since 7>1 after one wait)
* `-o-address-count` (number of opponent addresses participating in the analysis): 2 (if two opponent addresses are needed for the bug)

## Running a Single Benchmark

To build and run the analysis within one of the benchmark directories:

```bash
make          # compile the contract
make run      # run YulTracer on the compiled contract
```

The first step compiles the Solidity source with `solc`, generates and prepares the ABI, and runs all postprocessing scripts. The second step launches YulTracer on the entry-point contract.

## Running YulTracer on All Benchmarks

The script `run-yultracer-all.sh` runs YulTracer across all benchmark suites. It accepts `--timeout SECONDS` (default: 3600), `--out DIR` (default: `results`), and `--rerun-timeouts` (re-run benchmarks that previously timed out). On macOS it uses `caffeinate` to prevent sleep during long runs.

```bash
bash run-yultracer-all.sh                                     # default 3600s timeout, output under results/
```

This runs YulTracer on the `reentr-assert-vuln`, `reentr-assert-safe`, `reentr-vuln`, and `ac-vuln` benchmark suites and writes per-benchmark timing results under `results/`. Results are cached: re-running the script skips all previously completed benchmarks (including timeouts). Pass `--rerun-timeouts` to also re-run benchmarks that previously timed out.

Internally, the wrapper invokes `run-yultracer.py`, which can also be called directly for finer control:

```bash
python3 run-yultracer.py --root <benchmark-dir> --out <output-dir> [--timeout SECONDS] [--rerun-timeouts]
```

## Comparing YulTracer with Other Tools

To compare YulTracer against Mythril, Slither, and Solc's SMTChecker on the same benchmarks, first run each tool's wrapper script:

```bash
bash run-mythril-all.sh    [--timeout SECONDS] [--out DIR] [--rerun-timeouts]
bash run-slither-all.sh    [--timeout SECONDS] [--out DIR] [--rerun-timeouts]
bash run-smtchecker-all.sh [--timeout SECONDS] [--out DIR] [--rerun-timeouts]
```

**Note:** Running all tools end-to-end takes significant time. On the 23 `reentr-assert` benchmarks (vuln + safe), approximate total runtimes on a typical laptop are:

| Tool | Approximate total time |
|---|---|
| YulTracer | ~5 minutes |
| Slither | ~1 minute |
| Mythril | ~5 hours |
| SMTChecker CHC | ~11 hours |
| SMTChecker BMC | ~1 minute |

Results are cached, so interrupted runs can be resumed without re-running completed benchmarks. Pass `--rerun-timeouts` to also re-run benchmarks that previously timed out.

Each wrapper runs its tool on the relevant benchmark suites (`reentr-assert-vuln` and `reentr-assert-safe`; `run-smtchecker-all.sh` runs both CHC and BMC engines). Like the YulTracer wrapper, they accept `--timeout SECONDS` (default: 3600), `--out DIR` (default: `results`), and `--rerun-timeouts`, and store per-benchmark timings under `DIR/`.

Each wrapper invokes a corresponding Python runner script that can also be used directly:

* `run-mythril.py` — runs Mythril (`--modules` to select detector modules)
* `run-slither.py` — runs Slither (`--detect` to select detectors; default: `reentrancy-eth,reentrancy-no-eth,reentrancy-balance`)
* `run-smtchecker.py` — runs Solc's SMTChecker (`--engine {bmc,chc}`, with options for `--targets`, `--ext-calls`, `--invariants`)

Once results from all tools are available, use `comparison.py` to produce a combined comparison table:

```bash
python3 comparison.py results/reentr-assert
python3 comparison.py results/reentr-assert -o my-comparison.csv
```

This reads the `timings.csv` files from each tool's results subdirectory and produces a single CSV (default: `comparison.csv`) inside the given results directory.

The output CSV has one column per tool/variant combination (e.g. `mythril (v)`, `mythril (s)`, `yultracer (v)`, `yultracer (s)`, etc.), where `(v)` is the vulnerable variant and `(s)` is the safe variant. Each benchmark row shows one of:

* `\rejected MM:SS.T` — the tool reported a vulnerability, with the time taken
* `\passed MM:SS.T` — the tool did not report a vulnerability, with the time taken
* `t/o` — the tool timed out
* `N/A` — no result available for this tool/benchmark combination

After the benchmark rows, summary rows show:

* **Rejected / Passed / Timeout** — percentage of benchmarks in each category per tool
* **Average time** and **Total time** — computed over non-timed-out benchmarks

## Notes

The following benchmarks had to be modified:

- `reentrancy_modifier_reentrancy`: Original was safe in 0.5.0+. Pure annotation removed because it now compiles to `STATICCALL`, which does not allow state changes.
- `reentrancy_reentrancy_bonus`: Original was safe without a `deposit` method. Added `fallback` and logic to make the bug visible.
- `reentrancy_reentrancy_cross_function`: Original was safe without a `deposit` method. Added `deposit` method to make reentrant withdraw possible.
- `reentrancy_reentrancy_insecure`: Original was safe without a `deposit` method. Added `deposit` method to make reentrant withdraw possible.
- `reentrancy_modifier_reentrancy`: The benchmark does not send any ETH, so a YulTracer assertion hook was added to capture the reentrancy error on the token balance.
